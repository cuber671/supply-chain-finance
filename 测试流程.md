### 🛑 C2 — 遇错即停，只报不修 (Halt & Report)

一旦遇到以下任意情况， **立即停止所有后续步骤** ，不得执行任何后续操作，进入 Phase 3 故障诊断流程：

* HTTP 响应码与该步骤定义的 **预期响应码不符** （如预期 200 实际返回 4xx/5xx）
* 业务响应码与预期不符（如约定成功码 `code != 0`）
* 数据库断言不符（字段值、数据行数与预期不一致）
* 区块链交易 Revert、回执 `status != 1`或超时未上链
* 日志中出现**与当前请求 TraceID 强关联**的 `Exception`、`ERROR`、`FATAL`级别日志
* MQ 消息消费失败、缓存写入 / 读取异常、文件存储操作失败等分布式链路异常
* 幂等性校验失败（重复提交产生脏数据）

 **补充规则** ：WARN 级别日志仅记录不触发停测，仅当业务定义的致命 WARN（如数据一致性告警）且与当前 TraceID 关联时，纳入停测触发条件；日志断言必须严格匹配当前请求的全局唯一 TraceID，禁止全量扫描日志误触发。

### 🔁 C3 — 修复后必须重测失败步骤 (Re-test on Resume)

1. **先清理脏数据** ：清除该失败步骤产生的所有测试脏数据，评估对前置步骤的影响，若脏数据已破坏前置依赖状态，必须从受影响的最近前置步骤开始重跑，禁止直接重测当前步骤
2. **必须重新执行本次失败的步骤** ，不得跳过直接进入下一步
3. 重测结果无论通过与否，均须输出**重测记录条目**追加到《步骤执行全记录》，原始失败记录永久保留，严禁覆盖、删除、修改
4. 重测通过后方可继续后续步骤
5. 同步更新《接口覆盖追踪表》中对应条目的状态与执行次数

  -not -path "*/test/*" -not -path "*/target/*" -not -path "*/build/*" -not -path "*/.git/*" \
  | sort > ./test_output/controller_list.txt

# 2. 扫描类上的父路径@RequestMapping注解（排除注释）

  --include="*.java" --include="*.kt" \
  -not -path "*/test/*" -not -path "*/target/*" -not -path "*/build/*" -not -path "*/.git/*" \
  $(cat ./test_output/controller_list.txt) \
  | grep -v "//" | grep -v "/*" | grep -v "*/" | grep -v "class\|interface" \
  > ./test_output/class_path_mapping.txt

  -not -path "*/test/*" -not -path "*/target/*" -not -path "*/build/*" -not -path "*/.git/*" \

### 📝 Phase 1 — 制定 "业务闭环" 测试计划与覆盖率校验

#### 1.1 生成全量测试计划

基于 Phase 0 的接口原始清单，按 **业务生命周期强依赖顺序** （无依赖公共接口→用户域→权限域→基础数据域→业务核心域→交易域→清算域）排列所有接口，输出有序测试计划。 **计划步骤数必须 ≥ N，且每个 API-XXX 必须有且仅有一个对应 Step，核心写接口需额外补充幂等性测试 Step** 。

每个步骤必须包含以下完整信息：

* **Step ID** ：S01、S02……（严格有序，无跳号）
* **覆盖接口** ：对应的唯一枚举 ID（如 API-003），一个 Step 原则上仅覆盖一个接口
* **业务描述** ：该步骤的完整业务语义与测试目标
* **物理调用链路** ：`客户端 → 网关 → 服务名 → Controller#方法名`

* **产出变量** ：执行后需提取并传递给后续步骤的变量名、提取规则
* **脏数据清理规则** ：该步骤产生的测试数据对应的表名、清理条件
