### 步骤 3：严格按修复方案实施代码修改

1. **逐文件、逐修复点实施** ：按照修复方案的顺序，一次只修改一个文件、一个修复点，严禁批量修改多个文件
2. **修改前二次校验** ：每次修改前，再次用 `Read`工具读取目标文件的当前内容，确认与修复方案的「修改前代码」完全一致，避免代码版本不一致导致的改错
3. **精准修改** ：使用 `Edit`工具，严格按照修复方案的「修改后代码」，精准修改指定行号的内容，不得修改上下文无关代码
4. **修改后即时校验** ：每个修复点修改完成后，立即用 `Read`工具读取修改后的文件内容，与修复方案的「修改后代码」逐行比对，确保修改完全符合方案要求，无错改、漏改、多改
5. **单文件修改完成后** ：立即记录该文件的修改详情，写入 `test/changes/`目录，格式见下文输出规范

### 步骤 4：编译 / 语法校验（修改完成后立即执行）

1. 按照项目的技术栈，执行对应的编译 / 语法校验命令：
   * Java 项目：`mvn compile -DskipTests` 或 `gradle compileJava`
   * Go 项目：`go build ./...`
   * TypeScript 项目：`tsc --noEmit`

2. 校验结果处理：
   * ✅  **编译 / 语法无报错** ：进入下一步功能验证
   * ❌  **编译 / 语法报错** ：立即暂停，定位报错原因，若为修改导致的错误，立即反馈用户，给出修正建议，经用户确认后再调整；若为原有项目的前置报错，反馈用户，确认是否继续实施

## 六、异常场景处理规则
1.  **方案与代码不匹配**：实施前发现目标文件代码与修复方案的修改前代码不一致，立即暂停，反馈用户，确认正确的代码版本/调整修复方案，不得擅自修改方案内容
2.  **编译/语法报错**：修改后出现编译报错，立即暂停，定位报错根因，若为修改导致的错误，第一时间反馈用户，给出修正建议，经用户书面确认后再调整；若为项目原有报错，反馈用户，确认是否继续实施
3.  **功能验证不通过**：修复后原故障仍存在，立即暂停，重新分析根因，反馈用户，给出补充修复建议，经用户确认后再调整，不得擅自修改原修复方案
4.  **出现次生问题**：修复后正常业务流程出现回归问题，立即暂停，第一时间告知用户，同时提供回滚方案，经用户确认后可执行回滚，恢复至修改前状态
5.  **文件/路径不存在**：实施前发现修复方案指定的文件不存在，立即暂停，反馈用户，确认正确的文件路径，不得擅自修改其他文件
6.  **用户中途终止实施**：若用户在实施过程中发出终止指令，立即停止所有操作，若已完成部分修改，告知用户已修改的内容，并提供回滚方案

## 七、回滚执行规范
当用户发出回滚指令时，必须严格按照以下步骤执行，确保快速、安全恢复：
1.  确认回滚范围：明确用户需要回滚的实施编号、修改的文件列表
2.  确认备份文件：校验备份目录中的原文件是否完整、版本正确
3.  执行回滚操作：
```bash
# 确认备份目录
BACKUP_DIR="test/backup/[对应的备份目录]"
# 覆盖原文件，恢复至修改前版本
cp -f ${BACKUP_DIR}/[原文件路径] [目标文件原路径]
# 逐文件执行覆盖，确保所有修改的文件都恢复
```

4. 回滚后校验：执行编译 / 语法校验，确认无报错；重启服务，验证业务功能恢复正常
5. 输出回滚报告：将回滚执行过程、校验结果、恢复状态，写入 `test/report/fix-rollback-report-[实施编号].md`，同时在终端输出回滚结果
