## 补充诊断：数据库状态与 Flyway 脚本不一致

### 进一步根因分析

经过深入检查，发现：

1. **V20 脚本内容**（V20__init_sys_admin.sql 第20-50行）：
   - 创建 user_id=`1000000000000000001` 的 admin 用户
   - 密码哈希：`$2a$10$ToBidmYMQ4SbiEECQdww2uVpGkNMCMFPbb3mCBtSq.vNboQ92zPqa`（对应密码 123456）
   - 使用 `ON DUPLICATE KEY UPDATE`（基于 username 唯一索引）

2. **数据库实际状态**：
   - 存在 user_id=`1` 的 admin 用户（早期版本创建）
   - **不存在** user_id=`1000000000000000001` 的 admin 用户

3. **问题根因**：
   - V20 脚本的 `ON DUPLICATE KEY UPDATE` 基于 username 唯一索引
   - 当 username='admin' 重复时，应更新密码
   - 但数据库中 `user_id=1` 的记录**没有被更新**
   - 可能原因：Flyway baseline 时已存在 data=True 的记录，脚本未正确执行

4. **影响**：
   - 测试脚本使用密码 123456（符合 V20 脚本）
   - 但数据库中 admin 的密码并非 123456 的哈希
   - 导致登录验证永远失败

### 建议修复方案

需要数据库管理员执行以下操作之一：

**方案 A**：更新现有 admin 用户的密码
```sql
UPDATE t_user SET password='$2a$10$ToBidmYMQ4SbiEECQdww2uVpGkNMCMFPbb3mCBtSq.vNboQ92zPqa' WHERE user_id=1;
```

**方案 B**：删除早期记录，让 V20 脚本重新创建
```sql
DELETE FROM t_user WHERE user_id=1;
-- 然后重启 auth-service 触发 V20 重执行
```

**方案 C**：使用 V20 脚本创建的用户 ID 登录（如果存在）
```sql
SELECT * FROM t_user WHERE user_id=1000000000000000001;
```

### 问题定性
- **非测试脚本错误**：测试脚本使用密码 123456 是正确的
- **数据库状态问题**：早期数据与 Flyway 迁移脚本不一致
