# 第一章：核心数据实体目录

## 1.1 核心业务数据实体定义

以下为核心业务数据实体清单，每个实体均具有唯一ID、业务含义和状态变更特性。

### 1.1.1 用户与认证域

| 实体名                     | 数据库表            | 业务含义                                      | 核心标识字段      |
| -------------------------- | ------------------- | --------------------------------------------- | ----------------- |
| **User**             | t_user              | 平台用户（员工）主表，记录企业员工基本信息    | userId (雪花算法) |
| **Enterprise**       | t_enterprise        | 企业主表，记录平台所有企业主体的基本信息      | entId (雪花算法)  |
| **LoginTransaction** | t_login_transaction | 企业登录TCC分布式事务记录，用于追踪登录一致性 | txId (自增)       |

**User 实体状态机：**

```
STATUS_PENDING(1) -> STATUS_NORMAL(2) -> STATUS_FROZEN(3) -> STATUS_CANCELLING(4) -> STATUS_CANCELLED(5)
    (待审核)           (正常)           (冻结)           (注销中)            (已注销)
```

**Enterprise 实体状态机：**

```
STATUS_PENDING(0) -> STATUS_NORMAL(1) -> STATUS_FROZEN(2) -> STATUS_CANCELLING(3) -> STATUS_CANCELLED(4)
    (待审核)           (正常)           (冻结)            (注销中)            (已注销)
```

### 1.1.2 仓储服务域

| 实体名                        | 数据库表                | 业务含义                               | 核心标识字段 |
| ----------------------------- | ----------------------- | -------------------------------------- | ------------ |
| **WarehouseReceipt**    | t_warehouse_receipt     | 电子仓单主表，记录仓储货物的电子凭证   | id (自增)    |
| **StockOrder**          | t_stock_order           | 入库单主表，记录货物入库申请与确认流程 | id (自增)    |
| **Warehouse**           | t_warehouse             | 仓库主表，记录仓储企业的仓库信息       | id (自增)    |
| **ReceiptEndorsement**  | t_receipt_endorsement   | 仓单背书转让记录表                     | id (自增)    |
| **ReceiptOperationLog** | t_receipt_operation_log | 仓单拆分合并操作日志表                 | id (自增)    |

**WarehouseReceipt 实体状态机：**

```
STATUS_IN_STOCK(1) -> STATUS_PENDING_TRANSFER(2) -> STATUS_SPLIT_MERGED(3) -> STATUS_BURNED(4) -> STATUS_IN_TRANSIT(5) -> STATUS_VOID(6)
    (在库)              (待转让)                (拆分/合并)        (已核销)        (运输中)        (已作废)

ON_CHAIN_STATUS: PENDING(0) -> SYNCED(1) / FAILED(2)
```

**StockOrder 实体状态机：**

```
STATUS_PENDING(1) -> STATUS_CONFIRMED(2) -> STATUS_CANCELLED(3) -> STATUS_COMPLETED(4)
    (待确认)           (已确认)           (已取消)            (已完成出库)
```

### 1.1.4 金融服务域

| 实体名                    | 数据库表           | 业务含义                             | 核心标识字段 |
| ------------------------- | ------------------ | ------------------------------------ | ------------ |
| **Loan**            | t_loan             | 质押贷款主表，记录仓单质押贷款全流程 | id (自增)    |
| **LoanRepayment**   | t_loan_repayment   | 贷款还款记录表                       | id (自增)    |
| **LoanInstallment** | t_loan_installment | 贷款分期计划表                       | id (自增)    |
| **Receivable**      | t_receivable       | 电子应收款项主表，记录供应链应收账款 | id (自增)    |
| **RepaymentRecord** | t_repayment_record | 还款记录表（应收款还款）             | id (自增)    |

**Loan 实体状态机：**

```
STATUS_PENDING(1) -> STATUS_REJECTED(2) / STATUS_CANCELLED(3)
      |                   |
      v                   v
STATUS_PENDING_DISBURSE(4) -> STATUS_DISBURSED(5) -> STATUS_REPAYING(6) -> STATUS_SETTLED(7)
                                 |                        |
                                 v                        v
                              STATUS_OVERDUE(8) -> STATUS_DEFAULTED(9)
```

**Receivable 实体状态机：**

```
STATUS_PENDING(1) -> STATUS_ACTIVE(2) -> STATUS_PARTIAL_REPAYMENT(3) -> STATUS_SETTLED(4)
      |                |                    |
      v                v                    v
                   STATUS_OVERDUE(5) ------>+
```

## 1.2 实体关系图（文字版）

```
Enterprise (企业)
  ├── User (用户) [1:N, enterpriseId]
  ├── WarehouseReceipt (仓单) [1:N, ownerEntId]
  ├── Loan (贷款) [1:N, borrowerEntId]
  ├── Receivable (应收款) [N:M, creditorEntId/debtorEntId]
  └── EnterpriseCreditProfile (信用档案) [1:1, entId]

WarehouseReceipt (仓单)
  ├── StockOrder (入库单) [1:1, receiptId]
  ├── ReceiptEndorsement (背书) [1:N, receiptId]
  ├── Loan (质押贷款) [1:1, receiptId]
  └── LogisticsDelegate (物流) [1:N, receiptId]

LogisticsDelegate (物流委派单)
  ├── LogisticsTrack (轨迹) [1:N, voucherNo]
  └── Receivable (应收款) [1:1, sourceVoucherId]

Loan (贷款)
  ├── LoanRepayment (还款记录) [1:N, loanId]
  └── LoanInstallment (分期计划) [1:N, loanId]
```

## 场景 A：企业注册与审核上链

### 1. 业务链路总览

**终点：** 企业状态变为 STATUS_NORMAL(1)，区块链地址写入 t_enterprise.blockchain_address

# 第四章：架构与数据风险洞察及建议

## 4.1 数据一致性风险

### 风险点 1：仓单质押与贷款创建的强耦合

**描述：** 在"场景C"中，仓单锁定（lockReceipt）与 Loan 创建是强依赖关系，但两者在同一个本地事务中。如果先锁定仓单成功但 Loan 创建失败，需要手动补偿解锁仓单。

**涉及接口：**

- `LoanService.applyLoan()` -> 调用 `WarehouseReceiptService.lockReceipt()`
- `WarehouseReceiptService.lockReceipt()` -> 更新 WarehouseReceipt.isLocked=true

**建议：**

- 增加定时任务补偿机制：扫描 isLocked=true 但 loan_id 对应 Loan 不存在的异常状态
- 或引入 Saga 模式，将仓单锁定作为 Loan 创建的 Try 阶段

---

### 风险点 2：区块链上链失败的数据补偿缺失

**描述：** 多个场景（仓单签发、背书转让、贷款放款/还款、应收款生成/确认/还款）的区块链上链操作均为"弱依赖"，失败仅记录日志，无自动补偿机制。

**涉及接口：**

- `BlockchainFeignClient` 所有调用
- 典型问题：链上失败 -> 本地已生效 -> 数据不一致

**建议：**

- 增加 `t_blockchain_transaction_record` 补偿表
- 定时任务扫描：本地已变更但链上未成功的记录，自动重试上链
- 或使用消息队列（MQ）异步上链，本地变更后发送消息，消费失败进入死信队列

---

### 风险点 3：企业登录TCC事务的Confirm/Cancel漂移

**描述：** `AuthController.login()` 对企业登录使用 TCC 模式（EnterpriseLoginService.tryLogin/confirmLogin/cancelLogin），但 Confirm/Cancel 执行结果未知时依赖客户端轮询 `queryLoginStatus`，无服务端主动补偿。

**涉及接口：**

- `EnterpriseLoginService.tryLogin()` -> 创建 LoginTransaction
- `EnterpriseLoginService.confirmLogin()` -> 更新 LoginTransaction
- `EnterpriseLoginService.cancelLogin()` -> 更新 LoginTransaction

**建议：**

- 增加定时任务扫描：LoginTransaction.tryTime 超过阈值（如5分钟）仍未 Confirm/Cancel 的记录，主动触发查询或 Cancel
- 或将 TCC 模式替换为可靠消息模式（本地消息表 + 定时任务投递）

---

## 4.2 性能风险

### 风险点 4：物流轨迹上报频率带来的数据库压力

**描述：** `LogisticsController.reportTrack()` 定时上报 GPS 轨迹（如每30秒一次），高并发场景下 LogisticsTrack 表写入量巨大。

**涉及接口：**

- `LogisticsService.reportTrack()` -> Insert LogisticsTrack

**建议：**

- 轨迹数据写入使用 ClickHouse 或时序数据库（TDengine）替代 MySQL
- 或增加轨迹缓冲：先写入 Redis，批量落库

---

### 风险点 5：企业信用评分实时计算

**描述：** `CreditService.recalculateCreditLevel()` 每次触发时实时计算信用评分，可能涉及大量 CreditEvent 聚合计算。

**涉及接口：**

- `CreditService.recalculateCreditLevel()` -> 查询 CreditEvent 列表 -> 计算总分

**建议：**

- 信用评分采用异步计算：事件上报后投递消息，异步更新 EnterpriseCreditProfile
- 或增加缓存层：信用评分计算结果缓存5分钟

---

## 4.3 业务逻辑风险

### 风险点 6：仓单拆分合并的原子性保障

**描述：** `WarehouseReceiptController.executeSplitMerge()` 执行仓单拆分/合并时，需要原子性更新多个仓单（父仓单作废、子仓单创建/更新）。如果中途失败，可能导致数据状态不一致。

**涉及接口：**

- `WarehouseReceiptService.executeSplitMerge()` -> 多个 Update/Insert 操作

**建议：**

- 增加分布式事务或 Saga 模式保障
- 或使用"拆分合并锁"：创建 ReceiptOperationLog 作为操作记录，定时任务补偿检查

---

### 风险点 7：贷款逾期检测的及时性

**描述：** Loan 逾期状态（STATUS_OVERDUE）依赖定时任务检测（LoanOverdueCheckTask），检测间隔（如每日）可能导致逾期状态更新延迟。

**涉及接口：**

- `LoanOverdueCheckTask` -> 扫描 Loan 表 -> 更新 STATUS_OVERDUE

**建议：**

- 缩短检测间隔（如每小时）
- 或在用户访问贷款详情时实时计算是否逾期

---

### 风险点 8：信用额度超额发放

**描述：** `CreditService.setCreditLimit()` 设置授信额度后，如果 EnterpriseCreditProfile.usedLimit 计算不准确（如借款后未及时更新），可能导致超额授信。

**涉及接口：**

- `CreditService.setCreditLimit()` -> Update EnterpriseCreditProfile.availableLimit
- `LoanService.disburseLoan()` -> 更新 Loan 状态 -> 应同步更新 EnterpriseCreditProfile.usedLimit

**建议：**

- 每次放款/还款时，校验可用额度 = 授信额度 - 已用额度 - 本次金额 >= 0
- 增加 EnterpriseCreditProfile.usedLimit 的 ACID 保障

---

## 4.4 安全风险

### 风险点 9：仓单操作权限校验分散

**描述：** 多个仓单操作（背书、拆分、合并、核销）的权限校验逻辑分散在 Controller 层，未集中到 Service 层，可能存在校验遗漏。

**涉及接口：**

- `WarehouseReceiptController.launchEndorsement()` -> 手动校验 ownerEntId
- `WarehouseReceiptController.applySplit()` -> 手动校验 ownerEntId
- `WarehouseReceiptController.applyBurn()` -> 手动校验 ownerEntId

**建议：**

- 抽取 `WarehouseReceiptService.checkOwnership(receiptId, entId)` 统一权限校验
- 或使用 AOP 拦截 @RequireReceiptOwner 注解

---

### 风险点 10：区块链私钥安全

**描述：** Enterprise.encrypted_private_key 加密存储在数据库，但加解密密钥管理未在代码中明确体现，存在密钥泄露风险。

**涉及接口：**

- `EnterpriseService.registerEnterprise()` -> 生成密钥对
- 区块链签名操作 -> 解密私钥

**建议：**

- 使用硬件安全模块（HSM）管理私钥
- 或使用密钥分散存储（sharding）
