# API版本管理指南

## 当前版本状态

**当前版本**: `v1`
**基础路径**: `/api/v1/`

## 已实现

1. **URL版本控制**: 所有API使用`/api/v1/`前缀
2. **OpenAPI文档**: Swagger UI可访问，用于API文档和测试

## 版本升级策略

当需要破坏性API变更时，应创建`v2`版本：

### v1 -> v2 升级规则

1. **不兼容变更**（必须升级到v2）:
   - 删除或重命名端点
   - 更改必需参数
   - 更改参数类型
   - 更改响应结构

2. **兼容变更**（可在v1内迭代）:
   - 添加新的可选参数
   - 添加新的响应字段
   - 添加新的端点

### 升级步骤

1. 创建`/api/v2/`路径的新Controller
2. 将不兼容的端点迁移到v2
3. 在v1 Controller中添加@Deprecated注解
4. 更新Swagger文档
5. 通知客户端升级

## 示例：v2端点

当需要v2时，新的端点路径示例：
- `/api/v2/finance/loan`
- `/api/v2/warehouse/receipt`

## 客户端建议

1. 使用v1前缀的客户端应做好版本检测
2. 关注API变更公告
3. 优先使用可选参数而非必需参数（增加向前兼容性）
