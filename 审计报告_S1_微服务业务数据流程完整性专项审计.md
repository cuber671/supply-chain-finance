1. **区块链上链/链下数据不一致** (H2/H3): 应收账款调整和融资操作未上链，导致链上链下数据永久不一致

#### 3.1.3 场景内完整性缺陷明细

> 【风险等级】：高
> 【缺陷归属】：节点6-仓单作废、warehouse-service
> 【缺陷类型】：数据一致性与同步缺陷
> 【缺陷描述】：仓单作废操作`voidReceipt()`在执行区块链调用之前就先将DB中的仓单状态更新为BURNED。如果区块链调用失败，DB状态已经是BURNED，但链上状态不是，导致链上链下不一致
> 【代码位置】：WarehouseReceiptServiceImpl.java Lines 1204-1243
> ```java
> // Lines 1204-1206: DB状态先更新
> receipt.setStatus(WarehouseReceipt.STATUS_BURNED);
> receipt.setRemark("作废原因: " + reason);
> warehouseReceiptMapper.updateById(receipt);
>
> // Lines 1208-1243: 然后才执行区块链调用
> if (blockchainFeignClient != null && receipt.getOnChainId() != null) {
>     // blockchain call...
> }
> ```
> 【上下游影响范围】：影响仓单核销的链上链下一致性，可能导致链上显示"未核销"但链下已是"已核销"状态

> 【缺陷编号】：SC-001-02 (关联H7)
> 【风险等级】：高
> 【缺陷归属】：企业注销流程、enterprise-service
> 【缺陷类型】：数据内容缺陷
> 【缺陷描述】：企业注销时调用`checkAssetBalance()`查询资产余额，但该方法返回全零值（注释说"暂时设置为0，后续完善"）。这意味着即使企业有未结清的仓单、票据、应收账款，注销检查也会通过
> 【代码位置】：EnterpriseServiceImpl.java Lines 624-650
> ```java
> // 暂时设置为0，后续完善
> balance.setWarehouseReceiptCount(0);
> balance.setBillCount(0);
> balance.setReceivableCount(0);
> balance.setTotalAssets(0);
> ```
> 【上下游影响范围】：可能导致企业在有未结清资产时被错误注销，影响金融域和仓单域的关联数据

> ```
> 【上下游影响范围】：链上链下应收款金额永久不一致，监管审计时无法以链上数据为准

> 【缺陷编号】：SC-006-02 (H3)
> 【风险等级】：高
> 【缺陷归属】：节点6-应收款融资、finance-service

> 【上下游影响范围】：链上无法确认应收款融资状态，可能导致重复融资风险
